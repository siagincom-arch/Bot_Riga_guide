# Фаза M11: Оптимизация скорости ответа (Latency)

**Исполнитель:** Antigravity  
**Цель:** Снизить latency (время ответа) бота с >60 сек до комфортных значений (<5-10 сек до первого токена) за счет упрощения пайплайна, кэширования и внедрения стриминга стейт-машины RAG.

## Задачи:

### 1. Архитектура: Удаление `halluck_check`
Узел проверки на галлюцинации дублирует вызов LLM и удваивает/утраивает latency, повышая вероятность упереться в лимиты RPM у Gemini.
- **Действие:**
  - Удалить `src/rag/nodes/halluck_check.py`.
  - Удалить шаблон `src/rag/prompts/halluck.j2`.
  - В `src/rag/state.py` убрать флаги `needs_halluck_check` или связанные с ним state ключи.
  - В `src/rag/graph.py` изменить маршрутизацию: после `generate` граф должен идти в `END`. Убрать циклы retry по галлюцинациям.
  - В `generator.j2` добавить инструкцию, строго обязывающую писать только на основе контекста (grounding prompt) для компенсации убранной проверки.
- **Тесты:** Отредактировать интеграционные тесты `test_rag_graph.py`, чтобы они проходили без узла `halluck`.

### 2. Кэширование ответов (Semantic / Exact Cache)
Не нужно генерировать ответ заново, если запрос идентичен недавнему.
- **Действие:**
  - В `src/kb/models.py` создать модель данных кэша.
  - В `src/kb/store.py` (или отдельный файл `cache.py` на базе SQLite) реализовать простую таблицу кэширования, хранящую хэш запроса (id достопримечательности + входной язык/вопрос) и сгенерированный текст.
  - **Для фото:** Хэш можно считать от vision recognizer'а (place_id). Если пользователь прислал фото и мы распознали "Памятник Свободы", отдаем сгенерированный ответ из БД по ключу `photo:pamyatnik-svobody-riga`, не вызывая `generate`.
  - В `src/rag/graph.py` добавить узел `check_cache` (сразу перед `generate`). Если есть попадание в кэш — обход `generate` прямо в `END`. Добавить `cache_update` после `generate`.

### 3. Стриминг ответа в Telegram (Streaming API)
Создание UX "Печатает..." для скрашивания ожидания, так как Gemini может отдавать ответ чанками.
- **Действие:** 
  - В `src/llm/gemini.py` добавить метод `generate_content_stream_async()`. 
  - Реализовать периодический edit сообщения в Telegram (через python-telegram-bot). Обновлять сообщение `INTERIM_ACK_TMPL` раз в 1-1.5 секунды новыми токенами, чтобы не словить Telegram API 'Too Many Requests'.
  - Так как LangGraph поддерживает stream execution (`.astream_events` или `.astream`), интегрировать стриминг чанков узла `generate` в телеграм callback.

### 4. Устранение повторений фактов (Ещё легенда)
При нажатии "Ещё легенда" бот перефразирует старые факты, а должен выдавать принципиально новые.
- **Действие:**
  - В `generator.j2` добавить строгую директиву: при наличии `session_history`, не дублировать факты/сущности, упомянутые ранее.
  - В узле `retrieve` (или `text_search`), если это запрос типа `followup` (Ещё легенда), можно отфильтровывать чанки (passages), которые уже использовались, либо подавать их в LLM с соответствующим промптом "расскажи то, что не упомянуто в прошлой истории".

### 5. Логика кнопки "Что рядом"
При нажатии "Что рядом" под достопримечательностью бот просит геолокацию пользователя, а должен показывать объекты вокруг текущей достопримечательности.
- **Действие:**
  - В `src/bot/gateway.py` в обработчике `nearby:` (для строки `elif data.startswith("nearby:"):`) изменить логику: вместо ответа `NEARBY_REQUEST`, нужно сделать запрос к `KBStore` для получения координат `place_id`.
  - От этих координат (lat, lon) вызвать `geo_nearby` и вернуть список мест вокруг, не требуя локации от устройства.

- [ ] При повторном запросе одного и того же места логгер фиксирует "Cache hit".
